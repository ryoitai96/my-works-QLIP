## 1. エグゼクティブサマリー

### 1.1 システムのビジョン

**QLIP**（Quality Labor Inclusion Platform）は、My WORKSが展開する障害者雇用マネジメントサービスを支えるエンタープライズ向け **HR-ESGプラットフォーム** である。

既存のサテライトオフィス事業と、新規のFlower Lab（ロスフラワー活用B2Eギフト）事業を含む**全事業ラインを横断的に統合管理**し、障害者雇用における「量から質へ」のパラダイムシフトに対応する。

以下の3つの価値を統合的に実現する。

1. **戦力化エンジン** — 障害特性と業務のインテリジェントなマッチングにより、メンバーの能力を最大化する
2. **エシカル・オペレーション基盤** — 分散拠点での業務品質・メンタルヘルスを統合管理し、持続可能な就労を担保する
3. **コンプライアンス証明システム** — 厚生労働省「質の向上」指針に完全準拠したエビデンスを自動生成し、監査・ESGレポートに対応する

### 1.1.1 ブランド階層

```
QLIP by My WORKS（プラットフォームブランド）
  └── My WORKS事業の現場知見から生まれた
      障害者雇用マネジメントプラットフォーム
        ├── サテライトオフィス事業（既存モデル）
        │     └── クライアント企業ごとに多様な業務を提供
        ├── Flower Lab事業（B2Eギフト・新規モデル）
        │     └── ロスフラワーのアップサイクル × B2Eアニバーサリーギフト
        └── 将来の事業ライン（拡張可能）
```

QLIPは特定の事業モデルに依存せず、**あらゆる障害者雇用の業務形態**を統合管理できる汎用プラットフォームとして設計する。

### 1.1.2 UIデザイン統一規定

全画面共通のビジュアルアイデンティティを以下の通り統一する。

| 要素 | 規定 |
|------|------|
| ベースカラー | 白基調（#FFFFFF）＋ 十分な余白（padding 24px以上） |
| ブランドアクセントカラー | **#ffc000**（ゴールドイエロー）。ボタン、進捗バー、アクティブ状態などの強調要素に使用 |
| デザインスタイル | モダンB2B SaaS風。角丸（border-radius: 8〜12px）、影（shadow-sm〜md）、清潔感のあるフラットデザイン |
| フォント | 日本語: Noto Sans JP / 英数字: Inter。メンバー向けUIはフォントサイズ16px以上 |
| アイコン | Lucide React を標準アイコンセットとして採用 |
| メンバー向け特記 | ふりがな対応、表情アイコン多用、1画面1アクション原則を維持 |

### 1.2 解決する課題

- 属人的な支援ノウハウ（定着率99%）のスケーラブルなアルゴリズム化
- **既存サテライトオフィス事業のスプレッドシート運用からの脱却**と、蓄積データのQLIPへの移行
- 5〜10社・複数拠点の並行運用における品質の均一化（既存＋新規事業の横断管理）
- 2026年7月の法定雇用率2.7%引き上げ、および「質の向上」指針法定化への先行対応
- 農園型雇用代行ビジネスとの構造的差別化の可視化

### 1.3 初期スケーリング前提

| パラメータ | MVP初期値 | Phase 2目標 |

| クライアント企業数 | 5〜10社 | 30社 |

## 2. ビジネスコンテキストと制約条件

### 2.1 法規制要件

本システムの設計は、以下の法規制に完全準拠しなければならない。システムアーキテクチャのパラメータ層で吸収し、法改正時にコード変更なく対応できる設計を必須とする。

#### 2.1.1 障害者雇用促進法と法定雇用率

| 時期 | 法定雇用率 | システム対応 |
|------|-----------|-------------|
| 2025年現在 | 2.5% | パラメータとして保持 |
| 2026年7月〜 | 2.7% | 日付トリガーで自動切替 |
| 将来の改定 | 未定 | 管理画面から動的変更可能 |

カウント方式（1人/0.5人の特例、週10〜20時間の短時間労働者カウント等）もパラメータ化し、改定に即応する。

#### 2.1.2 厚生労働省「質の向上」5要素への対応マッピング

| 質の要素 | 対応モジュール | システムが生成するエビデンス |
|---------|--------------|------------------------|
| 能力発揮の十分な促進 | A: 特性マッピング＆タスクアサイン | スキルマッチ率、OJT履歴、習熟度推移グラフ |
| 成果の事業活動への十分活用 | D: サンクス通知、E: 受注管理 | ギフト納品実績、サンクス通知受領数、エンゲージメント指標 |
| 適正な雇用管理 | B: エシカル生産管理 | 労働時間ログ、合理的配慮記録、面談履歴 |
| 正当な評価と処遇への反映 | C: コンプライアンス証明 | 客観的評価ダッシュボード、昇給判定根拠データ |
| 能力発揮にふさわしい雇用の安定 | B: 予兆検知 | 定着率データ、メンタルスコア推移、離職リスクアラート履歴 |

#### 2.1.3 もにす認定・ESG開示への対応

- もにす認定申請に必要なデータ項目を調査し、レポートテンプレートとして実装する
- ISO 30414（人的資本開示）のフレームワークに沿った指標出力をサポートする
- フラワーロス削減量のCO2換算ロジックを組み込み、統合報告書素材として出力する

#### 2.1.4 個人情報保護法・要配慮個人情報

本システムが扱うデータには、障害種別、メンタルヘルス情報、服薬履歴など**要配慮個人情報**が含まれる。個人情報保護法の厳格な取得同意・目的外利用禁止に準拠したデータガバナンスを設計の根幹に据える（詳細は4章セキュリティ要件にて定義）。

### 2.2 競合差別化の構造

本システムは、農園型雇用代行ビジネスとの以下の構造的差異を**データとして証明**する機能を持つ。

| 評価軸 | 農園型（淘汰対象） | QLIP + Flower Lab |
|--------|------------------|-------------------|
| 成果物の事業貢献 | 農作物 → 廃棄・無配布 | B2Eギフト → HR施策の中核 |
| 雇用管理の可視化 | なし（ブラックボックス） | 全プロセスのデータトレーサビリティ |
| 是正ルート | なし | 内部通報＋第三者監査機能を実装 |
| ESG価値 | 限定的 | フラワーロス削減のCO2換算を自動生成 |

基幹システムとしての組織管理機能を、ロール別に提供する。

#### 3.3.1 スーパーアドミン（R01）向け管理画面

| 管理機能 | 概要 |
|---------|------|
| **企業管理（テナント）** | クライアント企業の一覧・詳細表示、契約プラン管理、テナント作成・編集・無効化。検索・フィルタリング付き一覧画面 |
| **個人管理（ユーザー）** | 全ユーザー（JC・メンバー・人事担当者等）の一覧・詳細表示。ロール付与・変更、アカウント有効/無効の切替。テナントをまたいだ横断検索 |

#### 3.3.2 クライアント企業（R05）向け管理画面

| 管理機能 | 概要 |
|---------|------|
| **メンバー管理** | 自社に所属するメンバーの一覧・詳細情報管理。障害特性（障害種別・等級）、年齢、性別等のプロファイル閲覧。**職務経歴書等のファイルアップロード機能**（PDF/Word/画像対応、暗号化保存） |
| **チーム設定** | マネージャーと一般メンバーを区別し、チーム（部門）に所属させる機能。**検索付きマルチセレクト（Combobox）**でメンバーをチームにアサインするUI。チーム単位でのレポート閲覧権限の自動適用 |
| **レポート閲覧** | 自社メンバーの集計レポート、アセスメント成長傾向、転換推薦情報の閲覧 |

### 3.4 業務フロー導線のシームレス統合

ロール別のログイン後のデフォルト遷移先と、画面間の自動導線を定義する。

**ログイン時のロール別自動リダイレクト**:

| ロール | ログイン後のデフォルト画面 |
|--------|------------------------|
| R01 スーパーアドミン | 全拠点ダッシュボード |
| R02 拠点マネージャー | 拠点ダッシュボード |
| R03 ジョブコーチ | メンバーコンディション一覧 |
| R04 メンバー | **日報入力画面**（未入力の場合）/ ホーム画面（入力済みの場合） |
| R05 クライアント人事 | KPIダッシュボード |

**画面間の自動遷移フロー（メンバー）**:

```
ログイン
  → 日報入力画面（未入力の場合）
    → 入力完了 → 花が咲くアニメーション
      → 自動でタスク一覧画面へ遷移
        → タスク完了
          → サンクスカード送信画面への導線を自動表示
            （「ありがとうを送りますか？」）
```

この導線設計により、メンバーの1日のワークフローが**日報→タスク遂行→感謝の循環**として自然に流れる体験を実現する。

### 4.2 マイクロサービス／モジュラー設計方針

各モジュール（A〜E）は**独立デプロイ可能なサービス**として設計する。ただし、MVP段階ではモノリポ内のモジュラーモノリスとして開発し、Phase 2以降で段階的にサービス分離する。

| 方針 | 詳細 |
|------|------|
| サービス間通信 | 同期: REST API（内部）/ 非同期: イベントバス（メッセージキュー） |
| データ所有権 | 各モジュールが自身のデータストアを論理的に所有（スキーマ分離） |
| 共有カーネル | 認証・通知・パラメータ管理は共通基盤として全モジュールから利用 |
| イベント駆動 | 「メンタルスコア低下検知」「タスク完了」「ギフト納品」等のドメインイベントを発行し、モジュール間の疎結合を維持 |

### 4.3 ゼロトラストセキュリティアーキテクチャ

#### 4.3.1 データ分類

| 分類 | 対象データ例 | 保護レベル |
|------|------------|-----------|
| **L4: 極秘** | メンタルヘルス詳細、服薬履歴、内部通報内容 | AES-256暗号化 + 行レベルセキュリティ + アクセス毎監査ログ |
| **L3: 機密** | 障害種別、特性プロファイル、バイタルスコア | AES-256暗号化 + RBAC |
| **L2: 社外秘** | タスク進捗、生産実績、受注データ | RBAC + テナント分離 |
| **L1: 一般** | SOPコンテンツ、公開レポートテンプレート | テナント分離 |

#### 4.3.2 セキュリティ実装要件

- **認証**: OAuth 2.0 + OIDC。メンバー向けにはPINコード＋顔認証（簡易生体認証）オプション
- **認可**: RBAC + ABAC（Attribute-Based）のハイブリッド。テナントID × ロール × データ分類の三次元制御
- **通信**: 全通信TLS 1.3以上。内部サービス間もmTLS
- **クライアント企業データ**: 従業員の生年月日等は**SHA-256ハッシュ化**して保持。平文データはAPI連携時の一時処理のみで永続化しない
- **監査ログ**: 全データアクセスをAppend-Onlyストアに記録。改ざん検知のためハッシュチェーン（ブロックチェーン的手法）を適用
- **データ分離**: マルチテナント設計においてRow-Level Security（RLS）を必須適用。テナント間のデータ漏洩を論理的に防止

#### 4.3.3 要配慮個人情報の取得・管理フロー

```
メンバー入職 → 利用目的の明示的説明（平易な言葉 + 図解）
    → 本人の書面同意取得（電子署名対応）
    → 同意範囲のシステム登録（同意管理テーブル）
    → 同意範囲内でのみデータ入力・閲覧が可能
    → 同意の撤回要求 → 対象データの論理削除 + 物理削除スケジュール登録
```

#### 5.1.2 個別特性プロファイル

メンバー登録時に以下の情報を構造化して入力する。入力はジョブコーチ（R03）が本人との面談を通じて行う。

**基本情報**
- 氏名、年齢、障害種別（身体/知的/精神/発達/重複）、障害等級
- 手帳種類、取得日、有効期限
- 就労経験、希望する業務領域

**特性アセスメント（5段階スコア）**
- 視覚認知力 / 聴覚認知力 / 手先の器用さ / 体力・持久力
- 反復作業耐性 / 変化対応力 / 対人コミュニケーション
- 集中持続時間（目安分数） / ストレス耐性
- 特記事項（自由記述）: 感覚過敏、こだわり、得意パターン等

**配慮事項**
- 通院スケジュール / 服薬タイミング（入力は任意、本人同意ベース）
- 環境配慮（騒音、照明、温度等の感覚過敏への対応）
- コミュニケーション配慮（指示の伝え方、フィードバック方法）

**プロファイルの更新ルール**: 初回アセスメント後、月次の面談時に更新。タスク習熟度データとの突合により、プロファイルの精度を継続的に向上させる。

#### 5.1.3 マッチングアルゴリズム

```
Input:  メンバー特性プロファイル + 当日のメンタルスコア
        + マイクロタスクDB + 当日の生産計画
Output: 推奨タスクアサイン案（優先度付きリスト）

ロジック:
  1. タスク要求スキルとメンバーの特性スコアのコサイン類似度を算出
  2. 当日のメンタルスコアに応じて認知負荷の高いタスクにペナルティを付与
  3. 直近のタスク実績（習熟度・完了率）で補正
  4. 同一タスクの連続日数が閾値を超えた場合、バリエーション提案を挿入
  5. 結果をジョブコーチのダッシュボードに「提案」として表示
     → JCが承認/修正 → メンバーのタブレットに今日のタスクとして配信
```

**重要**: アルゴリズムは「提案」であり、最終判断は必ずジョブコーチが行う。完全自動化はしない（人間の介在を設計原則とする）。

#### 5.1.4 視覚的SOP配信システム

- 各マイクロタスクに対応するSOP（Standard Operating Procedure）を、動画・写真・図解の3形式で登録可能
- メンバーのタブレット端末に、当日アサインされたタスクのSOPを自動配信
- LD（学習障害）等に配慮し、テキスト情報を最小化。ステップごとの画像遷移で工程を理解できるUI
- SOPの閲覧履歴をトラッキングし、習熟度判定の補助データとする

#### 5.1.5 定期アセスメント＆キャリア開発エンジン

メンバーの成長を定量的に可視化し、人事評価およびキャリアパス設計に接続する仕組み。松為(2006)の**職業準備性ピラミッド**を構造的基盤とし、NIVR「就労支援のためのアセスメントシート」の設問体系、デコボコベース社の5段階リッカート尺度アプローチを統合した**5領域アセスメントシステム**として設計する。

**設計の理論的基盤**:

| 参照フレームワーク | 取り入れる要素 |
|----------------|-------------|
| 職業準備性ピラミッド（松為, 2006） | 4層の階層構造（疾病・障害の管理→日常生活→職業生活→職務の遂行）。「土台が安定してこそ上層が伸びる」という発達的視点。**五角形パラメータ表示の構造的根拠** |
| NIVR アセスメントシート | 3領域の設問体系（作業遂行/職業生活/対人関係）、推奨＋選択の項目構造、自己評価＋協同評価、ストレングス特定、就労継続のための環境アセスメント（10領域） |
| デコボコベース社 アセスメント | 5段階リッカート尺度、感情コントロール・ストレス対処の深掘り設問群、定期的な定量トラッキング |

**雇用形態別の位置づけ**:

| 雇用形態 | 対象者 | アセスメントの活用目的 |
|---------|--------|---------------------|
| 有期雇用（パート・アルバイト） | 多数を占める | 成長トラッキング、戦力化の可視化、**正社員転換レディネス**の評価 |
| 無期雇用（正社員） | 一部 | 正式な**人事評価制度**への組み込み、昇給・昇格判定の客観的根拠 |

##### アセスメント体系: 職業準備性ピラミッド準拠 5領域

松為ピラミッドの4層にNIVR/デコボコの「対人関係・感情コントロール」を独立領域として加え、**5領域の五角形（ペンタゴン）パラメータ**で可視化する。

```
              ▲ 職務の遂行（D1）
             / \
            /   \
  対人関係・ /     \ 職業生活の
  感情      /       \ 遂行（D2）
  コントロール        
  （D3）   /    ★    \
          /           \
    日常生活 ─────── 疾病・障害
    の遂行（D4）      の管理（D5）

  ★ = ピラミッドの「土台→頂点」の関係を
      五角形で横断的に可視化
      D5が安定 → D4が安定 → D2,D3が伸びる → D1が開花
```

**回答尺度**: 5段階リッカート（NIVRのABC尺度を定量トラッキング向けに変換）
```
5: 非常に当てはまる  （NIVR A相当: 支援なしでできる）
4: やや当てはまる   （NIVR A〜B相当: おおむねできる）
3: どちらともいえない（NIVR B相当: 支援ありでだいたいできる）
2: あまり当てはまらない（NIVR B〜C相当: 支援ありでも不安定）
1: 全く当てはまらない （NIVR C相当: 支援ありでもできない）
追加: ?: 未経験・未確認・評価困難（NIVRの?に対応。スコア算出から除外）
追加: ★: ストレングスになる（NIVRの「ストレングスになる」に対応。該当項目にフラグ）
```

##### アセスメント実施フロー

```
実施タイミング:
  - 月次（簡易版）: 各領域から重点3問を抽出 × 5領域 = 計15問
  - 四半期（フル版）: 全5領域 × 各10問 = 計50問（ウィザード形式5ステップ）
  - 半年（環境版）: 就労継続のための環境アセスメント（10領域）
  - 随時: 配置転換時、新タスク習得完了時、面談時

実施方法（協同評価方式 — NIVR準拠）:

  Step 1: メンバー本人がタブレットUIで自己評価を入力
           → ウィザード形式: D1→D2→D3→D4→D5 の5ステップ（各10問/画面）
           → 進捗バー（#ffc000）で「あと何ステップか」を常時表示
           → 平易な日本語 + ふりがな付き + 読み上げ対応
           → 表情アイコン付き5段階ボタンで直感的に回答
           → ストレングスフラグ: 「これは得意！」ボタンで自己申告
           ※ 自己評価の段階では結果をJCに非表示（率直な回答を促進）
           → 完了後、即座に五角形チャートをアニメーション付きで表示

  Step 2: 担当ジョブコーチが協同評価を実施
           → 同じ設問に対し、2つの視点で評価:
             (a) 支援・配慮なしでの到達度
             (b) 支援・配慮ありでの到達度
           → 日次の観察記録・タスク実績データを参照しながら回答
           → ストレングスフラグ: JCが観察に基づき強みを特定

  Step 3: 自己評価と協同評価のギャップを自動分析 + 五角形パラメータ生成
           → 3種のギャップを可視化:
             ・自己評価 vs JC評価（配慮なし）→ 自己認知のズレ
             ・JC評価（配慮なし）vs JC評価（配慮あり）→ 支援の有効性
             ・前回スコア vs 今回スコア → 成長の方向性
           → 五角形チャートを自動生成（前回との重ね表示）
           → ピラミッド構造に基づくインサイトを自動生成
           → 双方が「ストレングス」としてフラグした項目を「確定ストレングス」として記録

  Step 4: 面談での協同確認（NIVRの「協同評価」の思想）
           → JCとメンバーが五角形チャートを一緒に確認
           → 「ここが広がったね」「ここが強みだね」のポジティブフィードバック
           → ピラミッド構造を踏まえた優先課題の合意（土台優先）
           → 次期の成長目標を1〜2つ設定（システムに記録）
           → 環境アセスメント実施時は合理的配慮の見直しも実施
```

**設計原則**: アセスメントは「テスト」ではなく「成長の振り返り」として位置づける。NIVRの「協同評価」の思想に則り、支援者と本人が対等な立場で「一緒に考える」プロセスとする。五角形チャートは「欠点の指摘」ではなく「形が広がる＝成長の実感」として体験できるUI設計とする。

### 5.2 モジュールB: エシカル生産管理＆メンタル・バイタル予兆検知エンジン

**目的**: 需要変動に対してメンバーの心身の安全を最優先に守りながら、納期と品質を担保する「エシカルな生産体制」をシステムで実現する。

#### 5.2.1 エシカル・クオータ（生産安全枠）

各メンバーに対し、心身に過度な負荷をかけない**月間最大安全生産量**を設定する。

**算出ロジック**:
```
エシカル・クオータ = 基礎処理能力 × 稼働可能日数 × コンディション係数 × 安全マージン(0.8)

基礎処理能力: 過去30日の平均タスク完了数（タスク種別ごとの重み付き）
稼働可能日数: 月間出勤予定日数 - 通院日 - 予定休暇
コンディション係数: 直近7日間のメンタルスコア平均 / 基準値(3.0)
安全マージン: 固定0.8（20%のバッファ。管理画面から調整可能）
```

- エシカル・クオータを超える生産アサインをシステムが**ブロック**する（JCが明示的にオーバーライドする場合は理由入力を必須とし、監査ログに記録）
- 拠点全体の生産キャパシティ = 各メンバーのエシカル・クオータの合計

#### 5.2.2 需要予測と負荷分散ダッシュボード

```
データフロー:
  クライアント企業HRIS → [CSV or API] → QLIP
    → 従業員の誕生月・入社月を集計
    → 月別ギフト需要を算出（プラン別の単価・個数を加味）
    → 全拠点の生産キャパシティと突合
    → 超過月を検知 → 以下のアクションを自動提案:
      (a) ドライフラワー等の保存可能品を前月に前倒し生産
      (b) 他拠点への負荷分散（拠点間のクオータ余力を可視化）
      (c) クライアントへのプラン調整提案（納品日の分散等）
```

**ダッシュボード表示項目**:
- 月別需要予測グラフ（6ヶ月先まで）
- 拠点別キャパシティ充足率（緑/黄/赤のヒートマップ）
- 超過アラート一覧と推奨アクション

#### 5.2.3 日報・コンディション管理（バイタル・メンタルスコアトラッキング）

現場で実際に使用されている**紙の日報**に即した詳細項目をデジタル化。単なる気分入力ではなく、JCが現場判断に必要とする情報を網羅する。

**毎朝の日報入力項目（メンバー用タブレットUI）**:

| カテゴリ | 項目 | 入力形式 | 必須/任意 |
|---------|------|---------|----------|

| **気分** | 一言コメント | 自由入力（音声入力対応）。例: 「昨日楽しいことがあった」 | 任意 |
| **睡眠** | 就寝時間 | 時刻ピッカー（例: 22:30） | 必須 |

**日報入力UIの設計原則**:
- 入力完了時にアニメーション付きポジティブフィードバック（花が咲くエフェクト等）

- 強制感を排除：未入力でもペナルティなし。JCへの「未入力アラート」で間接的にフォロー

#### 5.2.4 予兆検知アルゴリズムとアクションレコメンド

```
入力データ:
  - 直近7日間のメンタルスコア推移
  - 出勤パターン（遅刻・早退・欠勤の頻度変化）
  - タスク完了率の変化
  - 過去の離職・パニック事例のパターンデータ（匿名化された全メンバーの統計）

検知ロジック（Phase 1: ルールベース → Phase 2: MLモデル）:
  Phase 1:
    IF 睡眠スコア3日連続低下 AND 気分スコア2日連続3以下
      → アラートレベル: WARNING
    IF 欠勤2日 AND 前週に比べタスク完了率20%低下
      → アラートレベル: CRITICAL
  Phase 2:
    過去の離職・不調事例を教師データとし、7日以内の不調リスクを確率で算出
    → リスク60%以上: WARNING / 80%以上: CRITICAL

アクションレコメンド:
  WARNING:
    - 「本日は認知負荷の低いタスク（T002, T005）へ変更を推奨」
    - 「午後に15分の個別面談をスケジュール」
  CRITICAL:
    - 「本日の業務量を50%に軽減」
    - 「拠点マネージャーへエスカレーション」
    - 「通院日の前倒しを本人に提案」
```

### 5.3 モジュールC: コンプライアンス証明＆ESGレポート自動生成

**目的**: 厚生労働省の「質の向上」指針に対する準拠をデータで証明し、クライアント企業の監査対応・ESG開示を自動化する。

#### 5.3.1 データトレーサビリティ要件

| 要件 | 実装方針 |
|------|---------|
| 全操作の監査証跡 | 誰が・いつ・どのデータに・何をしたかをAppend-Onlyログに記録 |
| 改ざん防止 | ログエントリにハッシュチェーンを適用。前エントリのハッシュを次エントリに含める |
| 保持期間 | 法定保存期間（労働関連記録: 5年）+ 1年のバッファ = 6年 |
| データエクスポート | 監査時に指定期間のログをCSV/PDFでエクスポート可能 |

#### 5.3.2 KPIダッシュボード

**クライアント人事担当者（R05）向けダッシュボード**:

| KPIカテゴリ | 表示指標 | 更新頻度 |
|------------|---------|---------|
| 雇用状況 | 実雇用率、法定雇用率との差分、カウント内訳 | リアルタイム |
| 能力発揮 | タスク習熟度推移、スキルアップ率、OJT実施回数 | 週次 |
| 雇用管理 | 労働時間実績、合理的配慮実施件数、面談実施率 | 月次 |
| 評価・処遇 | 評価スコア分布、昇給実績 | 四半期 |
| 雇用安定 | 定着率（3ヶ月/6ヶ月/1年）、離職理由分析 | 月次 |
| B2E効果 | ギフト納品数、サンクス通知応答率、従業員満足度指標 | 月次 |

#### 5.3.3 レポート自動生成

| レポート種別 | 出力形式 | 対象読者 | 含まれるデータ |
|------------|---------|---------|-------------|
| 行政報告書 | PDF/Excel | 労働局、ハローワーク | 雇用状況、労働時間、配慮実施記録、研修履歴 |
| もにす認定申請書 | PDF | 厚生労働省 | 質の5要素に対応する全KPI。認定基準との適合度スコア |
| ESGレポート素材 | PDF/PowerPoint | IR部門、ESG評価機関 | フラワーロス削減量（kg）、CO2削減換算（t-CO2）、定着率、スキルアップ実績 |
| 人的資本開示（ISO 30414） | PDF/CSV | 統合報告書作成チーム | 障害者雇用に関連する人的資本指標 |
| 月次運用レポート | PDF | クライアント人事担当者 | 当月の生産実績、メンバー稼働状況、品質データ |

#### 5.3.4 フラワーロス削減量の算出ロジック

```
月間フラワーロス削減量(kg) = Σ(調達花材の重量)
  ※ 調達花材は全て「廃棄予定」であったことが前提

CO2削減換算(t-CO2) = フラワーロス削減量(kg) × 廃棄時CO2排出係数
  ※ 係数は環境省の廃棄物処理に伴うCO2排出係数を参照
  ※ 係数はパラメータ管理で更新可能
```

### 5.4 モジュールD: 是正ルート＆非同期コミュニケーション

**目的**: 行政が最も問題視する「是正されない構造」を排除し、安全なコミュニケーションチャネルを実装する。

#### 5.4.1 内部通報システム（セーフティネット）

**通報チャネル**:
- メンバーのタブレットUIに常時表示される「SOSボタン」（目立つが威圧的でないデザイン）
- テキスト入力 + 音声入力（文字入力が困難なメンバーへの配慮）
- 完全匿名オプション（通報者のIDをシステム上で不可逆に匿名化）

**エスカレーションフロー**:
```
メンバーがSOS送信
  → 通報内容のカテゴリ選択（ハラスメント / 業務過多 / 体調不良 / その他）
  → カテゴリに応じた宛先への自動ルーティング:
      - ハラスメント → 外部第三者機関（社労士等） + QLIPスーパーアドミン
      - 業務過多 → 拠点マネージャー + QLIPスーパーアドミン
      - 体調不良 → 担当ジョブコーチ + 拠点マネージャー
      - その他 → QLIPスーパーアドミン
  → 受理確認の自動返信（24時間以内の初動を約束）
  → 対応履歴の記録（クローズまでトラッキング）
```

**設計上の絶対原則**: 現場のジョブコーチや拠点マネージャーが加害者である可能性を想定し、彼らを**バイパス**して直接上位機関に通報できる経路を必ず確保する。

#### 5.4.2 第三者監査閲覧権限

- R07（第三者監査者）に対し、**読取専用**のポータルを提供
- 閲覧可能範囲: 労働時間ログ、面談記録、研修履歴、内部通報の対応状況（匿名化済み）、品質検査記録
- 閲覧不可範囲: メンバーの個別メンタルスコア（集計値のみ閲覧可）
- 閲覧時は必ず監査ログに記録。閲覧目的の入力を必須とする

#### 5.4.3 プッシュ型サンクス通知（社内SNSの代替）

**ユーザーフロー**:
```
1. Flower Labでギフトを制作 → 箱にQRコード（ユニークID）を印字
2. クライアント企業の従業員がギフトを受取り、QRコードを読取り
3. ブラウザでストーリーページが表示:
   - 「このフラワーギフトは、○○ Flower Lab のメンバーが
      ロスフラワーを活用して心を込めて制作しました」
   - フラワーロス削減への貢献量（このギフト1つで○g分の花材を救済）
   - 「ありがとう」ボタン（ワンタップ）
4. 「ありがとう」がタップされると:
   - 制作担当メンバーのダッシュボードにプッシュ通知
     「○○事業部のAさんから感謝の通知が届きました🌸」
   - 通知は表示のみ。返信義務なし（SNS的ストレスを排除）
5. サンクス通知の累計数はメンバーの成果指標として蓄積
```

**設計原則**:
- 受取者（R06）のアクションは「ありがとう」ボタンのワンタップのみ。アカウント登録不要
- メンバーの個人名は表示しない（拠点名のみ）。プライバシーを保護
- メンバー側の通知はオプトアウト可能（通知を受取りたくないメンバーへの配慮）

## 6. 既存データ移行要件（サテライトオフィス事業のQLIPオンボーディング）

### 6.1 移行対象データの棚卸し

現在スプレッドシートおよび属人的に管理されている以下のデータを、QLIPに移行する。

| 移行対象 | 現在の管理方法（推定） | QLIP移行先 | 優先度 |
|---------|---------------------|-----------|--------|
| クライアント企業一覧 | スプレッドシート | Tenant（テナント）テーブル | **最優先** |
| メンバー基本情報 | スプレッドシート | Member + CharacteristicProfile | **最優先** |
| メンバーの障害特性・配慮事項 | JCの記憶 + 一部メモ | CharacteristicProfile | **最優先**（JCへのヒアリングベースで入力） |
| 利用企業ごとの業務内容 | 属人的把握 | MicroTask（SOカテゴリ）テーブル | 高 |
| 面談記録・支援履歴 | なし or 個人メモ | ConsultationRecord | 高（過去分は概要のみ、以降はシステム入力） |
| 勤怠・労働時間 | スプレッドシート or 紙 | WorkTimeLog | 高 |
| 請求・契約情報 | スプレッドシート | Contract テーブル | 中 |

### 6.2 移行方針

- **Phase 0のPoC開始前**に、既存スプレッドシートのフォーマットを調査し、CSVインポート用のマッピングテンプレートを作成する
- QLIP管理画面に**CSVインポートウィザード**を実装（カラムマッピング → プレビュー → バリデーション → インポート確定）
- 属人的ノウハウ（JCの記憶にある特性情報）は、Phase 0期間中にJCとメンバーの面談を通じて特性プロファイルとしてシステムに入力する。この作業自体がQLIPの初期データ構築とJCのシステム習熟を兼ねる
- 過去の面談記録や支援履歴は完全移行を求めず、「QLIP導入以降」を正式なデータ蓄積開始点とする（過去分は概要サマリーのみ任意入力）

### 6.3 移行時のリスクと対策

| リスク | 対策 |
|--------|------|
| スプレッドシートのフォーマットが拠点ごとにバラバラ | インポートウィザードでカラムマッピングを柔軟に対応。標準テンプレートも配布 |
| JCの記憶に依存する特性情報の正確性 | 「初回アセスメント」として正式プロセス化。3ヶ月後に再アセスメントで精度を補正 |
| 移行期間中の二重管理負荷 | 最大2ヶ月の並行運用期間を設定。期間終了後はスプレッドシート運用を完全停止 |

## 8. UI/UX設計方針

### 8.1 メンバー向けUI（タブレット PWA）

**設計コンセプト**: 「毎朝開くのが楽しみになる、エンパワーメントアプリ」

業務管理だけでなく、**モチベーション向上**を目的とした設計。メンバーが「自分は成長している」「自分は必要とされている」と実感できる仕掛けを全面に配置する。

| 方針 | 具体策 |
|------|--------|
| ゲーミフィケーション | 日報入力完了で花が咲くアニメーション。タスク完了でポイント獲得（ポイントは称号やバッジに変換） |
| **ストリーク（🔥）バッジ** | 連続出勤日数、体調良好日数、日報連続入力日数をバッジとして表示。記録更新時に特別演出 |
| ポジティブフィードバック | タスク完了時に「ナイス！」等の肯定メッセージをランダム表示。サンクス通知受信時に特別演出 |
| **今日の一言エリア** | JCや人事担当者からの応援メッセージ、システムからの季節の挨拶や労いコメント（「今週もおつかれさま！」等）を表示する専用エリア |
| 監視感の排除 | 「管理されている」ではなく「サポートされている」と感じるトーン。トラッキングデータは「自分の成長記録」としてメンバー本人にも可視化 |
| 低リテラシー対応 | アイコンベースのナビゲーション。文字情報の最小化。主要操作は3タップ以内で完結 |
| パニック防止 | 予期しない画面遷移や警告音を排除。穏やかな配色（白基調＋ブランドカラー#ffc000のアクセント） |

### 8.2 ジョブコーチ向けUI（タブレット/PC）

**設計コンセプト**: 「朝一で全メンバーの状態を30秒で把握し、即座にアクションできるダッシュボード」

**ダッシュボード主要要素**:

- HRIS連携の設定画面（CSV or APIの切替、データマッピング）

### 8.4 ギフト受取者向けUI（スマートフォン Webミニアプリ）

**設計コンセプト**: 「QRコードを読み取って10秒で完結」

- ストーリーページは1スクロールで完結（ギフトの背景 → フラワーロス削減の意義 → 「ありがとう」ボタン）
- 「ありがとう」ボタンは大きく、直感的に押せるデザイン
- 完了後に「あなたの感謝が届きました」の確認メッセージ

## 9. MVP定義とフェーズ別ロードマップ

### 9.1 Phase 0: PoC（概念実証）— 3ヶ月

**目的**: コアコンセプトの技術検証と初期ユーザーフィードバック取得。並行して**既存サテライトオフィス事業のデータ移行**を実施。

| スコープ | 含む | 含まない |
|---------|------|---------|
| モジュールA | 特性プロファイル入力、マイクロタスクDB（Flower Lab + サテライトオフィス両方）、手動タスクアサイン | AIマッチングアルゴリズム |
| アセスメント | **全50問（5領域×10問）のウィザード形式UI**、自己評価、五角形パラメータ即時表示 | JC協同評価、環境アセスメント、レディネススコア、人事評価連動 |
| 日報（モジュールB） | **紙の日報に即した詳細入力**（体温・体調・気分・睡眠時間帯・食事・服薬・頓服）、手動アラート | エシカルクオータ自動算出、予兆検知 |
| モジュールD | サンクス通知（QR → ありがとうボタン → 通知）、タスク完了→サンクスカードの導線 | 内部通報システム |
| 設定・管理 | **企業管理（テナント一覧/詳細）、個人管理（ユーザー一覧/詳細）、チーム設定（Combobox UI）、ファイルアップロード** | 法定パラメータ動的更新 |
| 業務フロー | **ロール別ログインリダイレクト、日報→タスク→サンクスの自動遷移フロー** | — |
| メンバーUI | **エンパワーメント設計**（ストリーク🔥バッジ、今日の一言エリア、白基調＋#ffc000アクセント） | — |
| データ移行 | スプレッドシートからのCSVインポート、JCとの特性プロファイル初回入力 | 過去の面談履歴の完全移行 |
| インフラ | シングルテナント、基本的なRBAC | マルチテナント、ゼロトラスト |

**検証項目**:

## 10. リスクマトリクスと緩和策

| ID | リスク | 発生確率 | 影響度 | リスクレベル | 緩和策 |
|----|--------|---------|--------|------------|--------|
| R1 | 要配慮個人情報の漏洩 | 低 | 極大 | **高** | ゼロトラスト設計、AES-256暗号化、RLS、定期ペネトレーションテスト、インシデント対応手順の事前策定 |
| R2 | 管理トラッキングによるメンバーへの精神的負荷 | 中 | 大 | **高** | ゲーミフィケーションUI、入力の任意性確保、定期的なメンバーアンケートによるUI改善、「見守り」のトーン設計 |
| R3 | 法改正への追従遅延 | 中 | 大 | **高** | パラメータ管理サービスによるコードレス対応、法改正モニタリング体制の構築、社労士との定期的な制度レビュー |
| R4 | 拠点間の品質格差 | 中 | 中 | **中** | 統一SOP配信、QC検査のシステム化（チェックリスト + 写真記録）、拠点横断のQC指標ダッシュボード |
| R5 | クライアント企業のHRIS連携障壁 | 高 | 中 | **中** | CSV手動アップロードを常に代替手段として維持、連携テンプレートの提供、導入支援チームによるハンズオン |
| R6 | メンバーの朝の入力継続率の低下 | 中 | 中 | **中** | ゲーミフィケーション（ストリーク、バッジ）、JCからの声かけ運用ルール、入力項目の定期的な簡素化見直し |
| R7 | サンクス通知の応答率低迷 | 中 | 低 | **低** | QRコード同梱の物理的なカード演出の工夫、ストーリーページのA/Bテスト、ワンタップの徹底（登録不要） |
| R8 | 予兆検知の偽陽性によるJCの「アラート疲れ」 | 中 | 中 | **中** | Phase 1ではルールベースで保守的な閾値設定、Phase 2でML導入時に偽陽性率を継続モニタリング、アラートの重要度3段階化 |
